# Experiments — 테스트 및 실험

## 목적
각 도메인 로직의 정확성과 품질을 검증하는 테스트 케이스를 관리한다.
MVP 출시 전 핵심 플로우 전체를 반드시 통과해야 한다.

---

## 테스트 범위 (MVP 기준)

| 도메인 | 테스트 항목 | 통과 기준 |
|--------|------------|----------|
| recommendation | 기업 유형별 추천 정확도 | 상위 1위 정답률 80% |
| scoring | 점수 계산 일관성 | 동일 입력 → 동일 출력 |
| kpi | KPI 생성 품질 | 전문가 검수 통과율 70% |
| kpi | 물리 검증 정확도 | 수식 오류 0건 |
| planning | 계획서 초안 완성도 | 섹션 누락 0건 |
| expert | 매칭 정확도 | 분야 일치율 90% |

## 핵심 테스트 시나리오 (E2E)

### 시나리오 1: 예비창업 기업
```
입력: 예비창업 / 소재 / TRL 3 / 대구
기대 출력:
  - 추천 1위: 예비창업패키지
  - 선정 확률: 55~70%
  - KPI: 물성 기준 정량 목표 1개 이상
  - 계획서: 6개 섹션 완성
```

### 시나리오 2: 초기 기업 (3년차)
```
입력: 초기기업 / 자동차부품 / TRL 5 / 이노비즈 보유
기대 출력:
  - 추천 1위: 소부장 강소기업
  - 선정 확률: 65% 이상
  - KPI: 경량화 or 강도 관련 물리 검증 포함
```

### 시나리오 3: KPI 물리 검증 실패 케이스
```
입력: "철강 소재로 50% 이상 경량화"
기대 출력:
  - 물리 검증: 불가 판정
  - 보정 제안: "최대 15% 경량화 가능, 소재 변경 시 30%까지 가능"
```

---

## 실험 기록 원칙

1. 모든 프롬프트 변경 실험은 날짜 + 결과 기록
2. A/B 테스트 결과는 승자만 반영 (패자 이유도 기록)
3. 물리 수식 오류 발견 시 즉시 kpi 도메인 CLAUDE.md 업데이트

## Claude 작업 규칙

- 새 기능 추가 시 테스트 케이스 먼저 작성 후 구현
- 테스트 실패 시 해당 도메인 CLAUDE.md 검토 후 수정
- 프롬프트 실험 결과는 test_cases.md에 누적 기록
